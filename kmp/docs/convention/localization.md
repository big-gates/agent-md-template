# Localization Guide

> **트리거**: UI 텍스트를 추가·수정할 때, `strings.xml` 또는 `composeResources/`를 변경할 때 이 문서를 참조한다.

## 목적
앱 전반의 텍스트가 일관된 톤앤매너를 유지하고, 다국어 대응이 누락되지 않도록 하기 위한 지침이다.

## 1. 톤앤매너 (Tone & Manner)
- **친절하고 정중함:** 사용자에게 정보를 제공하거나 요청할 때 항상 친절하고 정중한 어조를 사용한다.
- **간결하고 명확함:** 모바일 화면의 특성상 문장은 최대한 간결하게 유지하며, 사용자가 해야 할 행동을 명확하게 전달한다.
- **신뢰감 있는 전문성:** 앱의 도메인에 맞게 사용자에게 신뢰감을 줄 수 있는 정확한 단어를 선택한다.

## 2. 언어별 스타일 가이드

### 한국어 (ko)
- **문장 종결 어미:** 해요체(~해요, ~에요, ~요)를 기본으로 사용한다. 너무 딱딱한 하십시오체나 반말은 지양한다.
- **명사형 종결 지양:** 버튼이나 레이블을 제외하고, 문장 형태의 텍스트에서는 명사형으로 문장을 끝내지 않는다. (예: "저장 완료" -> "저장했어요")
- **공백 및 문장 부호:** 문장 끝에는 마침표(.)를 사용하며, 감탄사(!)는 꼭 필요한 경우에만 제한적으로 사용한다.

### 영어 (en)
- **Sentence Case:** 제목과 버튼 텍스트를 제외하고 일반 문장에서는 첫 글자만 대문자로 시작하는 Sentence case를 사용한다.
- **Active Voice:** 수동태보다 능동태를 사용하여 사용자의 행동을 직접적으로 유도한다. (예: "The product was added" -> "Product added")
- **Contractions:** 친근한 느낌을 위해 일상적인 대화에서는 'Don't', 'I'm'과 같은 축약형을 사용할 수 있다.

## 3. 리소스 관리 규칙
- **하드코딩 금지:** 모든 UI 텍스트는 `composeResources`의 `strings.xml`에 정의해야 한다.
- **키 네이밍 컨벤션 (Snake Case):**
    - `버튼`: `btn_login`, `btn_cancel`
    - `힌트/플레이스홀더`: `hint_email`, `hint_password`
    - `에러 메시지`: `error_invalid_email`
    - `안내 메시지`: `msg_welcome`
    - `화면 제목`: `title_home`, `title_settings`
- **중복 정의 지양:** 공통적으로 사용되는 단어(확인, 취소, 저장 등)는 `core:designsystem`의 리소스를 우선 활용한다.

## 4. 에이전트 판단 규칙

| 상황 | 판단 |
|------|------|
| 새 화면 추가 | `title_*` 키로 화면 제목 추가 + 본문/버튼/에러 문자열 일괄 추가 |
| 버튼 텍스트 추가 | `btn_*` 키 사용. 공통 단어(확인, 취소, 저장)는 `core:designsystem` 리소스 재사용 |
| 에러 메시지 추가 | `error_*` 키 사용. 사용자가 이해할 수 있는 문장으로 작성 |
| 기존 문자열 수정 | 두 언어(ko, en) 모두 수정. 변수 포맷(`%s`, `%d`) 일치 확인 |
| Composable에서 텍스트 사용 | `stringResource(Res.string.key_name)` 사용. 하드코딩 금지 |

## 5. 다국어 설정 체크리스트
- [ ] 하드코딩된 문자열이 없는가?
- [ ] `values` (기본/영어)와 `values-ko` (한국어) 리소스가 모두 존재하는가?
- [ ] 변수가 포함된 문장의 경우 `%s`, `%d` 등을 적절히 사용했는가?
- [ ] 줄바꿈(`\n`)이 두 언어 모두에서 자연스럽게 배치되었는가?
- [ ] 키 네이밍이 Snake Case 컨벤션(`btn_*`, `error_*`, `msg_*` 등)을 따르는가?
- [ ] 공통 단어를 `core:designsystem` 리소스에서 재사용했는가?
